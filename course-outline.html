Claude Code — Getting Started
A 3-session course for IT support and Atlassian / monday.com implementers
3 sessions × 60 min After office hours Mixed Windows & macOS Accounts: most participants have Claude access
Session 1 — Foundation & First Run
60 min  ·  After office hours
Exit goal: every participant has Claude Code installed, understands the product ecosystem, and has run a real prompt inside VS Code — the environment they'll use for the rest of the course.
12 min
orientation
The Claude product map
Before touching any tool, give participants a clear mental model of the ecosystem. Confusion between products is the single biggest source of learner frustration — resolving it upfront saves time across all three sessions.
Most familiar
Claude.ai
The chat interface — where most of your team already works. General Q&A, drafting, summarising.
Connectors
One-click integrations built into Claude.ai — Atlassian, Google Drive, Gmail, Calendar. No setup beyond linking accounts. Connectors live here, not in Claude Code.
This course's focus
Claude Code
Terminal-based agentic coding assistant. Reads your actual project files and writes, runs, and edits code. Used via terminal or VS Code extension.
MCP (Model Context Protocol)
Claude Code's way of connecting to external tools — Jira, GitHub, databases — with full programmatic control. Introduced in Session 3.
For non-developers
Cowork ✦ new
Desktop app for non-developers. Automates file and task management without requiring a terminal. The bridge between Claude.ai and Claude Code.
Good to surface for participants who find the terminal intimidating — Cowork is their immediate alternative.
Teaching tip: draw a left-to-right spectrum — Claude.ai (easiest, least control) → Cowork (no-code desktop) → Claude Code (most powerful, terminal-based). Ask where participants currently sit and where they want to get to. One-liner for class: "Connectors are to Claude.ai what MCP is to Claude Code — same idea, different power level."
10 min
hands-on
Install Claude Code via terminal
Native installer — no Node.js required. Run in parallel for both OS groups. macOS: one curl command in Terminal. Windows: one command in PowerShell. Authenticate with existing Anthropic accounts. Verify with claude --version.
⚠ Windows: Git for Windows must be installed beforehand (see pre-session checklist). Keep VS Code's integrated terminal as the backup if PowerShell causes issues.
8 min
concept
Brief CLI orientation — just enough
Spend a few minutes in the terminal so participants understand what Claude Code is doing under the hood. Cover only the essentials: cd to navigate, claude to launch, /help to list slash commands, and /quit to exit. Run one live prompt against a real file. The goal is demystification, not proficiency — they'll work in VS Code from here.
Anchor this moment: "The terminal is where Claude Code lives. VS Code is just a friendlier window into the same thing."
10 min
hands-on new
Shift to VS Code — install the Claude Code extension
Open VS Code, go to the Extensions panel, search for "Claude Code," and install the official Anthropic extension. Once installed, the Claude Code panel appears in the sidebar. Walk participants through the layout: chat panel, file context indicator, and inline diff view. Authenticate if prompted (re-uses the same account).
Why VS Code from here: file explorer + syntax-highlighted output + integrated terminal + inline diff review in one window. Closer to how developers actually work, and significantly less intimidating for non-developers than a bare terminal.
15 min
hands-on
First real prompt — inside VS Code
Open a folder containing a real work file (a Jira export CSV, a Confluence draft, a config file). Use the Claude Code sidebar panel to ask it to summarise or explain the file's structure. Observe the inline diff review when Claude suggests changes. The goal is to feel the "it actually understands my file" moment — not a toy example.
5 min
wrap-up
Homework & preview
Homework: open a work folder in VS Code and run one Claude Code prompt on a file you actually use. Preview Session 2: we will build a script that automates a real IT support or Atlassian task — bring your Jira CSV.

Session 2 — Claude Code as a Coding Assistant
60 min  ·  After office hours
Exit goal: participants build a working script that solves a real IT or Atlassian task inside VS Code, and understand how to iterate when Claude's output isn't right the first time.
5 min
recap
Homework debrief
Quick share — what did you prompt Claude Code with? What surprised you? Surface any installation or VS Code extension issues not resolved in Session 1.
8 min
concept
How Claude Code reads your project
Explain context: Claude Code sees all files in the open VS Code folder. Introduce CLAUDE.md — a plain text file you write once to tell Claude about your project, its conventions, and your team's preferences. Demo: create a CLAUDE.md that describes a typical Atlassian project structure. Show how it reduces repetitive explanation in prompts. This is the "configure once, benefit always" moment.
In VS Code, CLAUDE.md appears as a regular file in the explorer — participants can see and edit it without leaving their workspace.
5 min
concept
/plan mode — think before you build
Before asking Claude Code to write or change any files, use /plan to ask it to outline its approach first. Claude describes what it intends to do — which files to create or modify, what logic to apply — without making any changes. You review the plan, redirect if needed, then proceed. For non-developers this is especially valuable: it builds trust, catches misunderstandings early, and makes the iteration loop feel transparent rather than like a black box. One-liner for class: "Plan first, build second."
/plan is also a token-efficiency tool — catching a wrong direction before Claude writes 100 lines of code saves far more tokens than /compact ever could. Carry this habit into the workshop that follows.
30 min
workshop
Build: IT support ticket report generator
Give Claude Code a CSV export of Jira or monday.com tickets (use the sample file distributed before the session) and ask it to write a Python script that categorises tickets by type, counts by assignee, and outputs a clean HTML report. Instructor demos the full iteration loop in VS Code first — /plan to outline the approach, prompt, review inline diff, accept or reject changes, re-run. Then participants build their own version.

Key learning inside the workshop: how to use /plan to steer Claude before any code is written, how to review Claude's proposed file changes in VS Code's diff view, and how to rephrase a prompt when output doesn't match intent.
Why this project: immediately useful, requires no prior coding knowledge to direct, produces a visible HTML output shareable the next morning, and naturally exercises the accept/reject diff review flow that VS Code makes accessible.
7 min
demo
Live demo — Claude Code meets the Jira / monday.com API
Instructor-only live demo. Using a pre-authenticated API token, show Claude Code fetching real data directly from Jira or monday.com — no CSV, no copy-paste. Ask it to pull open tickets assigned to a specific user, or summarise board items by status. The point is the contrast: "Earlier you fed Claude a file. Here, Claude goes and gets the data itself."

Frame this explicitly as the bridge to Session 3: MCP automates and deepens exactly this pattern. Keep credentials and config out of participant hands for now — this is a demo, not a hands-on exercise.
Instructor prep: prepare a read-only API token before the session. Jira: Settings → Personal Access Tokens. monday.com: Profile → Developers → My Access Tokens. Test the connection beforehand — a live demo that fails on auth kills momentum.
5 min
concept
Token efficiency — first principles
Introduce the context window with a simple analogy: Claude Code's short-term working memory. Show what happens when a session grows too long. Teach two habits: /compact (summarise and continue) and /clear (start fresh for a new task). Connect to cost — fewer tokens used = lower API spend per session. Note that VS Code's Claude Code panel shows session length visually, which helps.
Instructor prep before Session 2: Prepare a sample Jira or monday.com CSV export (10–20 rows, real column names, fully anonymised) and share via a team folder. Participants use this file in the workshop so no one is blocked by data access or sensitivity concerns during the session. Also prepare a read-only API token for the live demo at the end of the session.

Session 3 — Power Use, Skills & Independent Learning
60 min  ·  After office hours
Exit goal: participants know role-specific use cases for their Atlassian and monday.com work, can create team skills, have seen MCP in action, and leave with a personal self-study roadmap.
5 min
recap
Workshop wins
Quick share of what the ticket report script produced. Surface any blockers from between sessions. Celebrate what worked — this anchors confidence before introducing more advanced material.
15 min
demo
Role-specific use cases
Tailored live demos for their actual tools, run directly from VS Code:

Atlassian: generate JQL queries from plain English descriptions, draft Confluence page structures from bullet-point notes, write Jira automation rule conditions in plain text.

monday.com: formula column logic, API webhook scripts, board configuration documentation.

Frame these as "second brain" patterns — Claude Code as the colleague who never forgets syntax or API structure.
15 min
concept
Claude's Skills (SKILL.md)
Introduce SKILL.md files — reusable instruction sets stored in your project that teach Claude Code how to handle your team's specific workflows. Live demo inside VS Code: create a skill file for "generate a Jira epic from a project brief." Participants then draft a skill for one task they do repeatedly. This is the bridge between a personal tool and a shareable team asset — check the file into your repo and the whole team benefits.
10 min
conceptnew
MCP — connecting Claude Code to your tools
Brief, demo-first introduction. Show Claude Code asking Jira for a real ticket by ID using the Atlassian MCP server — no copying, no pasting, Claude Code acts directly inside Jira. Explain the mental model: MCP is what lets Claude Code take actions inside your tools, not just generate text for you to paste.

Keep configuration detail out of scope for this session — frame it as "this is what's possible, here's where to go next." Point to docs.claude.com/en/docs/claude-code for the MCP setup guide.
Contrast clearly with Session 1's product map: Connectors (Claude.ai) = one-click, no config. MCP (Claude Code) = manual setup, full programmatic control. Same concept, different power level.
8 min
concept
Token efficiency — advanced habits
Introduce .claudeignore to exclude large or irrelevant folders (attachments, exports, archives) from Claude Code's context. Show how a well-scoped VS Code workspace costs fewer tokens and produces more focused output. Revisit CLAUDE.md as a token-efficiency tool — explicit instructions prevent Claude from spending tokens guessing your intent.
7 min
orientation
How to keep learning independently
Point to docs.claude.com for Claude Code and MCP documentation. Recommend the flywheel pattern: find a repetitive task → ask Claude Code to automate it → iterate until right → turn it into a SKILL.md → share with the team. Close with the framing: Claude Code is a learning accelerator — every script it writes is also an explanation you can study.
5 min
wrap-up
Personal commitment
Each participant states one task they will automate using Claude Code in the next two weeks — written down. Creates accountability without pressure and gives you a natural follow-up conversation starter.
